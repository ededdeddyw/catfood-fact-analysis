# (②③) 網羅性の定義 と アフィリ遮断

> このサービスの「製品人格」を決める2つの論点。
> 結論：**評価しない・順位を付けない・母集団を宣言する**。病院DBと完全に地続き。
> 前提・全体思想は `README.md`、悩み機能の設計は `01`、データ検証は `02` を参照。

# ② 網羅性の定義 — 「全網羅」を捨てて「宣言した母集団へのカバー率」にする

## 0. これは何の話？（非エンジニア向け）

「うちのサイトは漏れなく載せています」をどう証明するか。
病院は**行政の届出名簿**があるので「名簿の○○件中△△件を掲載」と証明できた。
フードには**全商品の公式リストが存在しない**。だから別の方法で「範囲」を証明する必要がある。

## 1. 問題

- 病院：行政名簿（届出施設）＝公的な母集団 → 「掲載漏れがない」を客観証明できる。
- フード：「日本で売られている全キャットフードの公式リスト」が**存在しない** → 同じ証明ができない。

## 2. 解き方：母集団を"宣言"して、それに対するカバー率を出す

MVVの網羅性の定義は「全部持っている」ではなく **「掲載範囲が客観的に証明できる」**。
フードでは **再現可能な収録ルール（母集団の境界）を先に宣言し、その中のカバー率と未掲載を正直に報告する。**
→ 病院の「行政名簿照合 315/1404 ＋ 未マッチレポート」と同じ証明構造。

## 3. 母集団の候補（＝フード版・行政名簿）優先順

| 候補 | 何の証明になるか | 評価 |
|------|-----------------|------|
| **ペットフード公正取引協議会の加盟ブランド** | 表示ルールを守るブランドの準公式リスト。**最も「行政名簿」に近い** | ◎ 第一母集団 |
| **愛玩動物用飼料安全法の届出 製造・輸入業者**（農水省／環境省） | フードを国内流通させる「会社」の母集団を画定（製品単位でなく上限を縛る） | ○ 会社レベルの網羅証明 |
| **主要EC（楽天/Amazon/大手ペットEC）の売れ筋上位N** | 「実際に買われているもの」を画定。ランキング自体が証明 | ○ 実需カバレッジ用 |

## 4. 設計：2層の母集団 ＋ カバー率レポート

1. **第一母集団（証明できる中核）** ＝ 公正取引協議会 加盟ブランドの**総合栄養食**。
   → 「対象母集団 ○○件中、当サイト掲載 △△件（カバー率%）」を病院と同じ形で公開。
2. **第二母集団（実需カバレッジ）** ＝ 主要ECの売れ筋上位。
   → 「よく売れているのに未掲載」を**未掲載リストとして自動レポート**（病院の未網羅レポートと同型）。
3. **除外の宣言** ＝ 最初は「総合栄養食」に限定し、一般食・おやつ・療法食は母集団外と明示。
   → 「網羅していない」ではなく「**最初からこの範囲だけを対象と宣言している**」に変わる。

## 5. MVV的に正しい言い方

- ❌「日本のキャットフードを網羅」
- ⭕「**ペットフード公正取引協議会加盟ブランドの総合栄養食 ○○件のうち △△件を、出典付きで掲載**（カバー率%・未掲載リストは公開）」

## 6. 病院との対応

| 病院 | フード |
|------|--------|
| 行政名簿（届出施設）= 母集団 | 公正取引協議会 加盟ブランド = 母集団 |
| 照合率 315/1404 を公開 | カバー率 △△/○○ を公開 |
| 未マッチ施設レポート | 未掲載商品レポート |
| 都届出名簿は開示請求で補完 | 会社届出リスト・EC売れ筋で補完 |

# ③ アフィリ遮断 — 「報酬で並び順が動かない」を構造で担保する

## 0. これは何の話？（非エンジニア向け）

キャットフードは紹介手数料が商品ごとに大きく違う。
「手数料の高い商品を上に出したい」誘惑に負けると、結局ふつうのランキングサイトになり、
このサービスの存在意義（ファクトで中立）が消える。それを**仕組みで防ぐ**。

## 1. 問題

フードの紹介手数料は商品ごとに桁違い。**高報酬商品を上に出したい誘惑が病院の送客より遥かに強い。**
負けたら「結局ランキングサイトの一種」＝参入の意味が消える。

## 2. 担保①：そもそもランキング機能を作らない ← 堀と防火壁を兼ねる

敵＝アフィリのランキング記事。**ランキングを持たないこと自体が差別化であり、同時にアフィリ汚染の構造的遮断**。
「おすすめ度」「総合○位」を**機能として実装しない**（＝`01` の「点数化しない」と一致）。

## 3. 担保②：表示順はマネー無関係の決定的ルールだけ

並び順は**ユーザーが選ぶ客観ソート**に限定：商品名順／価格順／カロリー密度順／開示の多い順。
→ どれも**手数料額を一切参照しない。**

## 4. 担保③：購入リンクは全商品・全チャネル平等

- 「購入先を見る」は**全商品に等しく付ける**（高報酬だけリンク、はしない）。
- 楽天／Amazon／公式など**複数チャネルを並べる** ＝ 価格比較ユーティリティであって誘導ファネルにしない。
- リンクは「どこで買えるか」の便益であり「これを買え」の推奨ではない。

## 5. 担保④：エディトリアル・ファイアウォール（コードで分離）

病院の `affiliate_status` を表示ロジックと分離したのと同じ思想を**コードレベルで強制**：

- ファクトDB（成分・出典・4状態）を生成する処理は、**手数料データに一切アクセスしない。**
- 並び替え・絞り込みのコードが手数料列を参照したら**テストで落とす**（lint / CI ガード）。
- 手数料情報は別テーブルに隔離し、表示順の入力に絶対しない。

## 6. 担保⑤：ユーザーへの開示文言（免責と同じ）

> 「当サイトは購入リンクから手数料を得る場合がありますが、**掲載順・内容には一切影響しません**。
>   当サイトは評価・ランキングを行わず、出典付きのファクトのみを掲載しています。」

## 7. 病院との対応

| 病院 | フード |
|------|--------|
| 送客パートナーと病院ファクトをデザイン/文言で分離 | 購入リンクとファクトを分離 |
| `affiliate_status=approved` のみ・表示順は無関係 | 手数料は別テーブル・表示順は無関係 |
| 口コミ評価をやらない | ランキング／おすすめ度をやらない |

# ②と③は相互補強する（重要な気づき）

- **③でランキングを捨てる**と、「結局どれが一番いいの？」という**網羅性への過剰圧力も同時に消える**
  （順位を付けないから「全部入れて比べる」必要がない）。
- **②で母集団を宣言する**と、「未掲載」が**手数料目当ての恣意的取捨ではなく、宣言した範囲に対する正直な差分**になる。
- 2つ合わせて「**評価しない・順位を付けない・範囲を宣言する**」という、病院DBと完全に地続きの製品人格が固まる。
