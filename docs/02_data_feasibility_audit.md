# (b) データ取得可否オーディット — 作る前に開示率を測る

> `01_kidney_consult_sheet.md` は**理想の出力**。
> でもリンを公式開示しているメーカーが2割しかなければ、腎臓シートの主役列が空欄だらけで成立しない。
> **作る前に50商品を手作業で調べ、各項目の開示率を実測する。** これは病院プロジェクトの
> 「掲載ファクト取得率スナップショット（§16-I）」をフードに移植したもの。

---

## 0. これは何をする作業？（非エンジニア向け）

実際に主要キャットフード50商品の公式サイトを開いて、
「リンの量は書いてある？」「カロリーは？」を1つずつ◯×でチェックする。
→ 「どの数値なら確実に集まるか」が分かる。
→ それで「腎臓シートは作れるか／看板を体重管理に変えるか」を**データに基づいて**決める。

所要：1人で2〜3日（50商品 × 各10項目を目視確認）。コストはほぼゼロ（API不要）。

---

## 1. ステップ1：50商品の選び方（層化サンプリング）

「売れ筋トップ50」ではなく、**開示率の現実が正しく見える**ように層を散らす。

| 層 | 配分の目安 | 検証したい仮説 |
|----|-----------|----------------|
| 価格帯 | 高級／中位／量販 各 約1/3 | 「安いフードほど開示しない」 |
| 形態 | ドライ／ウェット 両方 | ウェットの分析値開示率は別物 |
| 区分 | 総合栄養食／一般食／療法食 | 療法食はリン開示率が高いはず |
| 出自 | 国産／輸入 | 輸入は日本語表示と原文で差が出る |

> 偏ったサンプル（高級ばかり等）だと開示率を過大評価してしまう。層を必ず混ぜる。

---

## 2. ステップ2：1商品ごとに「開示されているか」を記録

`01` の列＝そのままチェック項目。**値そのものより「公式に載っているか」をまず数える。**

### 記録列（CSVスキーマ案）

```
product_id, maker, brand, price_tier, form, category,
  phosphorus_disclosed, protein_disclosed, sodium_disclosed, moisture_disclosed,
  calorie_disclosed, ingredients_disclosed, is_therapeutic,
  primary_source_type, source_url, fetched_at, notes
```

### 記録ルール

- `*_disclosed` は **◯（公式に数値あり）／△（範囲・曖昧）／×（記載なし）** の3値。
- **`primary_source_type`（一次ソース種別）の優先順位**（病院と同じ。混ぜない）：
  1. メーカー公式サイト
  2. パッケージ画像（公式）
  3. 法定表示（裏面ラベル）
  4. ECページ ← 最後の手段。公式が無いときのみ、出典として明記
- `source_url` と `fetched_at`（取得日）は**必須**。

---

## 3. ステップ3：開示率マトリクスを出す（意思決定の材料）

50件を集計して、項目ごとの開示率を出す。

```
項目          開示率(50件中)   想定判定
水分          100%            ◎ 確実に作れる（法定表示）
カロリー密度   ~90%?           ◎ 主力にできる
原材料        ~95%?           ◎
タンパク質    ~90%?           ○
ナトリウム    ~40%?           △ 列は出すが空欄多め前提
リン          ~25%?           ⚠ 腎臓シートの根幹が弱い
```

※ %は仮の予想値。**実測で置き換える。** 病院の §16-I と同じく、確定した数字だけ残し古い予想は消す。

---

## 4. ステップ4：撤退・修正ラインを先に決める（最重要）

数字を見てから決めると「データが無いのに無理やり載せる＝捏造」に流れる。**先に基準を置く。**

| リン開示率 | 判断 |
|-----------|------|
| **≥ 40%** | 腎臓シートは「開示している商品だけ」で成立。**GO** |
| **20〜40%** | 「リンを開示しているフード一覧」という**開示そのものを売りにする縮小版**で出す（README アイデア2を主役に） |
| **< 20%** | 腎臓は第3段でも見送り。**体重管理（カロリー密度）を看板**に切り替え |

→ どの結果でも「次に何を作るか」が決まる設計。空振りしても判断材料が残る。

---

## 5. ステップ5：副産物 — 乾物量換算の動作確認

水分・各成分が揃った商品を数件選び、実際に **ウェット↔ドライを乾物量基準に換算**してみる。

- 換算後の比較が「直感に反していないか」を目視確認。
- 例：水分の多いウェットは生表示だと栄養値が低く見えるが、乾物量換算で逆転するか。
- ここが機能すれば、サービスの技術的旗印（README）が早期に立証できる。

---

## 6. このオーディットのアウトプット

1. **開示率マトリクス**（どの列が作れるか）
2. **腎臓シートの GO / 縮小 / 見送り 判断**
3. **乾物量換算が機能するかの確認結果**
4. 副産物：そのまま MVP の最初の50商品データの種になる（手作業分が無駄にならない）

---

## 7. MVVとの整合

- 「件数だけ増やす処理はしない」→ 開示されていない数値は載せない。空欄は4状態で正直に。
- 「過大評価を書かない」→ 取得率は実測値で出す。予想で盛らない。
- 病院の段階主義（50→200→全件）と同じく、**まず50で現実を見てから拡張**する。
